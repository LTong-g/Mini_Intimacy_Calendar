# Developer Guide

## 0. 项目体积事实
- 仓库体积偏大的主要来源通常不是业务源码或 Git 历史，而是本机 Android 构建产生的可再生成中间产物。
- 常见占用路径包括 `node_modules`、`android/app/build`、`android/app/.cxx`、`android/build`、`dist` 和本机 Gradle 缓存目录。
- `node_modules` 中的主要占用通常来自 React Native / Expo 原生模块的 Android 构建目录，例如 `react-native-reanimated/android/build`、`expo-modules-core/android/build`、`react-native-screens/android/build`、`react-native-gesture-handler/android/build` 及对应 `.cxx` 目录。
- `android` 中的主要占用来自 `android/app/build` 和 `android/app/.cxx`。
- 上述 `build` 和 `.cxx` 目录属于可再生成构建产物，清理后下次 Android 构建会重新生成；`dist` 中的 Release APK 和导出资产属于发布/分发归档，是否保留取决于发布管理需要。

## 1. 项目定位
`MinimalistWeaponEnhancementCalendar` 是一个基于 Expo + React Native 的移动端打卡应用。  
它围绕“按日期记录三类行为”这个核心目标，提供日历视图、数据统计和数据导入导出能力。
三类行为均为代称，实际意义是三种应当自律，合理控制频率的行为。即软件功能为记录没有自律的时候，并提供一些统计分析。

## 2. 功能总览
### 2.1 日历主流程
- `日视图`：查看某一天的记录状态、图标信息、间隔天数，支持点击顶部日期文字用滚轮快速切换日期。
- `月视图`：按网格展示某月每天的打卡状态，支持左右滑动切月，也支持点击顶部月份文字用滚轮快速切换年月。
- 月视图点击当月日期时先选中日期，再次点击已选中的非未来日期会进入对应日期的日视图；点击已选中的未来日期不跳转；点击非未来的灰色日期会切到该日期所在月份并选中该日期；点击未来的灰色日期无操作。
- `年视图`：以 12 个月缩略日历展示全年分布，支持左右滑动切年，也支持点击顶部年份文字用滚轮快速切换年份。
- 底部栏左侧日历按钮按日视图、月视图、年视图轮回切换，按钮文字显示下一步可查看的视图；月视图和年视图的中间主按钮不提供返回日视图功能。

### 2.2 打卡记录
- 三类打卡项：
- `tutorial`（观看教程）
- `weapon`（武器强化）
- `duo`（双人练习）
- 软件内部统一使用“按日期记录次数”的新格式。
- 同一天、同一类型支持多次记录。
- 未来日期不可编辑。
- 底部打卡面板短按某类型会将该类型次数加 `1` 并关闭面板。
- 日视图已有记录图标长按时会按秒减少该类型次数，达到连续操作阈值后打开次数编辑弹窗。
- 日视图中同类型次数大于 `1` 时，图标内部显示次数数字。
- 月视图底部打卡面板长按某类型时直接打开次数编辑弹窗。
- 次数编辑弹窗使用滚轮选择次数；观看教程存在黑名单自动记录下限时，滚轮最小值为自动记录次数。
- 月视图与年视图按“某类型当天是否出现”展示状态，不展示同类型次数强度。

### 2.3 统计分析
- `总览统计`：总计、年均、分年统计。
- `年度统计`：指定年份总计、月均、逐月数据；顶部年份文字可点击弹出滚轮快速切换年份；当前年份的图表保留 1-12 月坐标轴但不绘制未来月份 0 数据，表格不显示未来月份，月均按已到月份计算。
- `自定义区间统计`：按日期区间聚合统计。
- 统计页在总览、年度和自定义区间模式下均先显示图表，再显示表格。
- 自定义区间进入日期选择页时，可点击顶部年份或年月文字用滚轮快速切换年份或年月，最终日期仍通过点击具体日期选定。
- 日期选择页从年份进入月份、从月份返回年份时使用与外层年份选择页进出一致的默认进入/退出动画；返回操作会先回到年份选择，选定具体日期后才返回来源页面。
- 自定义区间进入月统计时按真实年月倒序展示；跨年区间的月份标签显示为 `YYYY年M月`，避免不同年份的同名月份合并或错序。
- 统计口径按新格式中的次数累计，不再仅按每天每类型 `0/1` 计数。
- 折线图支持触摸查看点位数据；同一触摸点下多个项目数值相同时，数值标签按观看教程、武器强化、双人练习顺序逐个向右错开。
- 自定义统计图发生聚合时，触摸聚合点会隐藏其他横轴刻度和标签，仅显示该聚合点所在区间的起止日期；默认横轴只在末尾补充的终点标签与前一个标签按实际渲染测量宽度发生重叠时隐藏倒数第二个标签；触摸态若两个日期标签重叠，标签会自适应分散，首个聚合区间仅移动终点标签，末个聚合区间仅移动起点标签。

### 2.4 数据管理
- 设置页以 `数据管理` 框承载导入、导出和分享按钮。
- `导出`：导出包含打卡、黑名单和用户配置的本地数据 JSON，并可保存或分享。
- `导入`：从 JSON 恢复本地数据并做格式校验；新版完整备份恢复打卡、黑名单和用户配置，旧版打卡备份只导入打卡记录并保留现有黑名单数据。
- Android 原生构建中，应用发生 JS 未捕获异常、未处理 Promise、Android 未捕获异常或主线程长时间无响应时，会在应用专属外部文件目录的 `diagnostic-logs` 文件夹写入问题日志；正常运行不持续记录日志。
- 设置页仅在检测到本机已有问题日志文件时，在页面底部显示打开问题日志文件夹入口。

### 2.5 使用记录辅助
- 使用记录辅助为 Android-only 功能。
- 用户在设置页打开使用记录辅助开关后，应用会引导用户到系统设置手动授予使用情况访问权限，并在返回应用后按实际授权结果同步本地刷新计划。
- 使用记录辅助开关显示状态必须与系统使用情况访问权限同步；系统权限已授权时显示为开启，系统权限未授权时显示为关闭。
- 用户点击使用记录辅助开关时，界面先按用户目标状态乐观切换并跳转系统权限页；从系统权限页返回后再读取实际权限状态修正开关显示和刷新计划。
- 当系统使用情况访问权限被撤销或失效时，应用会在刷新权限状态时自动关闭应用内使用记录辅助功能并取消刷新计划。
- Android 不允许应用自行静默授予或撤销 `PACKAGE_USAGE_STATS`；用户关闭使用记录辅助开关时，应用必须跳转到系统使用情况访问权限页，返回后重新读取实际权限状态。
- 关闭使用记录辅助开关不会自动删除已经保存的应用使用记录；删除需要用户在开关关闭后点击设置页中的删除应用使用记录按钮。
- 删除应用使用记录按钮在开关开启时置灰不可交互，在开关关闭时恢复可交互。
- 为提高晚间刷新可靠性，应用会提供忽略电池优化和精确定时权限的开关样式入口；电池优化开关关闭时跳转到系统电池优化设置页，返回后重新读取实际状态。
- 开关开启时，应用会安排每天 23:55、23:56、23:57、23:58、23:59 各尝试读取并保存一次最近三天的黑名单应用使用时间段。
- 黑名单入口位于主界面右下角更多菜单的统计入口下方，只有使用记录辅助开关开启时可进入；黑名单页内不再提供系统授权按钮。
- 黑名单功能在独立黑名单应用设置页列出可启动应用并选择当前生效的黑名单应用，统计页只显示设置入口和当前已选择数量。
- 黑名单完整应用列表只在黑名单应用设置页读取；应用启动后首次进入该页时全量读取并缓存，本次运行期间再次进入优先使用缓存，右下角刷新按钮会重新全量读取。
- 黑名单应用设置页顶部已选择数量以原有边框卡片样式固定显示，卡片下方提供应用名称和包名搜索栏，搜索结果优先显示应用名匹配项且开头匹配排在前面，搜索栏下方左侧提供“全部 / 已选”列表切换；只滚动下方应用列表；应用列表中英文按拼音首字母混排，无法归入 A-Z 的应用归入末尾 #；应用列表框与上方卡片和搜索框同宽，首字母快速滑动条位于列表框外并贴近屏幕右侧边缘，触碰时高亮当前命中字母；右下角刷新按钮触发重新读取应用列表时，列表区域显示加载动画，刷新按钮自身不显示旋转动画。
- 黑名单应用设置页的勾选和取消勾选会实时保存为黑名单生效时间段，避免页面未正常退出时丢失已选择应用。
- 黑名单应用设置页会显示应用图标、应用名称和包名，便于从大量应用中快速识别目标；原生模块会基于应用图标提取主色并随按包名维护的应用档案保存，应用名、图标或主色变化时更新同一包名档案。
- 黑名单应用设置页刷新当前可启动应用列表时，会将当前生效黑名单中已卸载或不可启动的应用自动结束黑名单生效时间段并标记为已卸载，但不删除应用档案和历史使用记录。
- 黑名单统计页只读取使用情况和已保存黑名单应用档案，不自行读取完整应用列表或提取主色。
- 黑名单功能在独立使用时间段页面查看记录；页面先显示全部记录入口和按历史黑名单应用分组的入口，进入后显示对应记录列表。
- 黑名单使用时间段页面的全部记录详情中，每条记录前方显示对应应用图标。
- 黑名单使用时间段页面只读取已保存使用时间段和已保存黑名单应用档案，不读取完整应用列表或提取主色；全部记录详情和单个应用详情的顶部汇总固定显示，只滚动下方记录，记录上方可通过今日、7 天、30 天三个按钮切换范围且默认显示 7 天，右侧筛选按钮可打开筛选弹窗设置日期范围。自定义日期范围确认后，原范围按钮位置显示当前日期范围；再次打开筛选弹窗时显示当前筛选态，存在筛选时左侧操作为清除筛选。全部记录详情的筛选弹窗顶部按屏幕宽度显示当前选中应用图标预览，最右侧编辑按钮可打开应用显示筛选弹窗，在历史上加入过黑名单或已有使用记录的应用之间点击移动；单个应用详情的筛选弹窗顶部居中显示当前应用图标；记录详情按日期分组显示，日期标题右侧显示该日期合计使用时长。
- 黑名单功能支持读取黑名单应用使用时间段，黑名单主页在“查看使用时间段”入口下方通过“今日 / 7天 / 30天 / N周”按钮组切换显示当日、最近 7 天、最近 30 天统计图和自适应热力图，其中 N 周按当前屏幕实际可显示的热图周数动态更新。
- 黑名单主页热力图按 GitHub 贡献图形式展示使用时长，一列 7 个格子表示周日到周六，横轴只显示月份；图表按屏幕决定外框和绘图区宽高，再由一列 7 格约束计算格子大小，并由格子大小和绘图区宽度决定可显示列数。
- 黑名单主页热力图无使用格子使用白色或中性底色与中性边框，避免和黄橙色使用强度混淆。
- 黑名单主页热力图按每日使用分钟数连续渐变映射颜色，映射范围为 `0` 到当前可见范围单日最大分钟数向上取整到 10 的值；底部偏右显示对应的连续渐变图例，图例需与右侧边缘保持内缩，避免最大值位数较多时超出绘图区。
- 黑名单主页每次进入时会在后台静默读取最近三天内处于黑名单生效时间段的应用使用时间段，不显示动画或弹窗；下拉刷新使用同样的最近三天读取范围并显示刷新动画和完成弹窗；按日期读取记录按钮会弹出起止日期弹窗，点击起止日期框后复用自定义统计页的 DatePicker 页面选择逻辑和顺序校验，返回后继续显示读取弹窗，确认后只读取并保存所选日期范围内当时处于黑名单状态的应用记录，并在应用内完成弹窗中显示请求读取范围、实际读取到的记录覆盖范围和聚合后的使用时间段数量。
- 使用记录辅助开启后的晚间定时刷新与黑名单主页下拉刷新使用同样的最近三天读取范围，并会将读取到且命中黑名单生效时间段的应用使用时间段保存到本地。
- 黑名单应用内静默刷新、下拉刷新、按日期读取和晚间定时刷新统一通过 Android 原生刷新事务执行；原生侧负责选取目标范围内的黑名单生效包名、查询 UsageEvents、按生效时间段裁剪、合并碎片记录并写入本地黑名单数据，JS 侧只负责发起刷新、读取保存后的数据和展示结果。
- 黑名单功能将读取到且命中黑名单生效时间段的使用时间段保存为本地数据，包含包名、开始时间、结束时间和使用时长；取消黑名单或卸载应用不会删除历史使用记录。
- 黑名单功能会将同一应用中前一条结束时间与后一条开始时间间隔不超过 2 分钟、且间隔中没有其他黑名单应用使用时间段的碎片记录合并为一条使用时间段。
- 黑名单功能三个图表的绘图数据均为使用时长；当日饼图总范围为今天已经过去的分钟数，并将未使用黑名单应用的已过时间作为剩余扇区显示。
- 黑名单功能最近 7 天柱状图触摸时按坐标命中日期，框选对应柱子并在旁边显示时长，不使用 30 天趋势图那种垂直虚线。
- 黑名单功能最近 30 天趋势图中，连续多天无使用记录时只显示这段连续空心点的首尾点，中间仅保留趋势线；触摸图表时显示对应日期的竖向参考线、点位和分钟数。
- 黑名单功能最近 7 天柱状图和最近 30 天趋势图的纵坐标刻度标签显示在图表左侧。
- 黑名单功能最近 7 天柱状图和最近 30 天趋势图通过横轴数据范围左右内缩避免纵坐标刻度贴近绘图，不通过扩大绘图区左侧预留宽度实现。
- 黑名单功能记录的实际含义归入“观看教程”，因此黑名单主页、黑名单应用页、使用时间段页及图表中的按钮、筛选、选中态、刷新入口、弹窗内容和非白色视觉强调统一使用观看教程的黄橙色系；弹窗外界遮罩属于中性行为，不使用黄橙色系；当日饼图不使用应用图标主色。
- 黑名单使用记录会动态计算观看教程自动记录次数，并作为日历展示和统计中观看教程次数的下限；黑名单主页顶部提示框说明该联动，提示框内信息图标弹窗显示完整计算规则；用户通过 UI 修改的是观看教程总次数，但不能低于自动记录次数；尝试取消自动记录时可从提示弹窗跳转到使用时间段的全部记录详情，并自动筛选到触发提示的日期。
- 日历视图内触发的自动记录不可取消提示使用普通应用弹窗样式，不使用黑名单功能的黄橙色系。
- 自动记录次数基于已保存的黑名单使用时间段动态计算：在既有 2 分钟碎片合并结果基础上，所有黑名单应用使用时间段按时间排序后，前一段结束到后一段开始间隔不超过 1 小时的记录链视为同一次；聚合允许跨日，跨日聚合段按开始时间所在日期计入；同一次自动记录的已保存使用时长合计少于 5 分钟时不计入自动记录次数。

### 2.6 关于信息
- `关于`入口位于主界面右下角更多菜单最下方。
- `关于`：展示应用图标和当前语义版本。
- `软件介绍`：展示应用定位、三种记录类型名称和主要功能，面向用户时只介绍功能，不暴露权限、刷新、数据结构或实现规则等不必要细节。
- `使用帮助`：展示日视图、月视图、年视图、统计页、设置页和关于页的基础操作方式。
- `隐私政策`：说明基础记录数据、使用记录辅助权限、本地存储、导入导出分享和用户控制方式；应用不提供账号、云同步或主动上传记录数据的能力。

## 3. 数据模型与规则
### 3.1 存储结构
- 存储：`AsyncStorage`
- 对外导入、导出和分享使用完整应用数据 JSON，顶层包含 `schemaVersion`、`checkins`、`blacklist`、`settings` 和 `meta`。
- 本地运行期按顶层领域拆分存储，避免读取日历圆点时解析黑名单应用图标和使用记录等大数据。
- 当前本地键名：
- `app_data_checkins`：打卡记录。
- `app_data_blacklist`：黑名单应用档案、黑名单生效时间段、黑名单使用记录和刷新状态。
- `app_data_settings`：用户配置。
- `app_data_meta`：数据结构版本和迁移信息。
- 旧键名 `checkin_status`、`experimental_usage_blacklist` 和 `experimental_usage_intervals` 仅作为自动迁移来源保留。

对外备份结构示例：
```json
{
  "schemaVersion": 2,
  "checkins": {
    "recordsByDate": {
      "2026-04-29": {
        "tutorial": 2,
        "weapon": 1,
        "duo": 0
      }
    }
  },
  "blacklist": {
    "appsByPackage": {},
    "periods": [],
    "intervals": [],
    "refreshState": {}
  },
  "settings": {
    "usageAccess": {},
    "ui": {}
  },
  "meta": {}
}
```

### 3.2 打卡记录结构
- 结构：`checkins.recordsByDate` 中按 `{ "YYYY-MM-DD": { "tutorial": number, "weapon": number, "duo": number } }` 保存。
- 三个字段均为非负整数。
- 三个字段均为 `0` 的日期不落库。
- `tutorial` 字段保存用户设置的总次数；有效展示和统计时会与黑名单使用记录动态计算出的观看教程自动记录下限取较大值。
- 旧格式 `{"YYYY-MM-DD": number}` 仅在读取迁移和旧版打卡备份导入边界兼容。

示例：
```json
{
  "2026-04-29": {
    "tutorial": 2,
    "weapon": 1,
    "duo": 0
  },
  "2026-04-30": {
    "tutorial": 0,
    "weapon": 1,
    "duo": 1
  },
  "2026-05-01": {
    "tutorial": 1,
    "weapon": 1,
    "duo": 1
  }
}
```

### 3.3 黑名单记录结构
- `blacklist.appsByPackage` 按包名维护唯一应用档案，保存包名、应用名、图标、主色、首次看到时间、最近看到时间和当前安装状态。
- `blacklist.periods` 保存黑名单生效时间段，包含包名、生效开始时间、结束时间和结束原因。
- `blacklist.intervals` 保存已命中黑名单生效时间段的使用记录，包含包名、开始时间、结束时间和使用时长。
- 观看教程自动记录下限不单独持久化，读取日历和统计数据时从 `blacklist.intervals` 动态计算。
- 应用名、图标或主色变化时更新同一包名应用档案，不保存多个历史快照。
- 应用取消黑名单或卸载时只结束当前黑名单生效时间段，不删除应用档案或历史使用记录。

### 3.4 位掩码
- `1`：tutorial
- `2`：weapon
- `4`：duo
- 位掩码仅作为历史兼容和旧接口返回语义保留。

常见组合：
- `0`：无记录
- `3`：tutorial + weapon
- `5`：tutorial + duo
- `6`：weapon + duo
- `7`：三项全选

### 3.5 校验约束
- 日期键必须为 `YYYY-MM-DD`。
- 新格式值必须是对象，且字段只允许 `tutorial`、`weapon`、`duo`。
- 字段值必须是非负整数。
- 旧格式数值 `0..7` 仅在迁移和导入时兼容。
- 禁止未来日期修改。
- 新版完整备份导入必须通过顶层结构与各领域数据校验，并在导入时统一转换为当前 schema。
- 旧版打卡备份导入只校验并写入打卡记录，不清空现有黑名单数据和用户配置。

## 8. 发布版本管理与统一工作流
### 8.1 版本定义位置
- 语义版本（`appVersion`）定义在：
- `package.json` -> `version`
- `app.json` -> `expo.version`
- 当前语义版本（截至 `2026-05-09`）：`1.4.0`。

### 8.2 统一策略
- `runtimeVersion` 统一使用 `policy: appVersion`（iOS/Android 一致）。
- 构建号统一由 EAS 远端管理：
- `eas.json` -> `cli.appVersionSource: remote`
- `eas.json` -> `build.production.autoIncrement: true`
- 发布前通过脚本同步远端版本状态到本地：
- `npm run release:sync-version`（执行 `eas build:version:sync`）。

### 8.3 版本更新时在哪里改
- 仅在发布新业务版本时修改语义版本：
- `package.json` 的 `version`
- `app.json` 的 `expo.version`
- 不手工维护 Android `versionCode`（由 EAS 递增与同步）。

### 8.4 推荐发布流程（Android）
1. 修改语义版本（示例：`1.4.0 -> 1.4.1`）。
2. 执行 `npm run release:sync-version`。
3. 执行 `eas build --platform android --profile production`。
4. 归档产物时带上语义版本与构建号（示例：`MinimalistWeaponEnhancementCalendar-v1.4.0+42-android.apk`）。

### 8.5 约束说明
- `expo run:android` 用于开发调试，不作为发布产物来源。
- 发布产物统一来自 EAS `production` profile（云构建或 `--local`）。

## 9. Git 提交防事故流程
适用场景：工作区存在多项改动，且本次只允许提交其中一部分。

### 9.0 提交类型约定
- `[feat]`：仅用于应用内用户可见功能新增或交互行为新增。
- `[dev]`：用于开发流程、构建/发布链路、工程脚本、环境与测试工作流改造等非用户可见能力变更。
- `[fix]`：用于缺陷修复（用户可见或开发链路问题均可）。
- `[doc]`：用于文档更新。

### 9.1 提交前检查
1. 执行 `git status --short`，确认全部改动清单。
2. 执行 `git diff -- <目标文件>`，确认目标改动边界。
3. 若目标文件混有无关改动，先备份该文件（例如 `Copy-Item <file> <file>.bak`）。

### 9.2 构造最小提交
1. 只暂存目标文件：`git add <目标文件列表>`。
2. 提交信息按仓库规范填写（英文 `summary` + 逐文件 `description`）。
3. 完成提交后，若之前为隔离提交临时清理过文件内容，需把无关改动恢复回工作区未提交状态。

### 9.3 提交后复核
1. 执行 `git show --name-only --stat -1`，确认提交文件范围正确。
2. 执行 `git status --short`，确认无关改动仍在工作区且未被误提交/误丢失。
3. 若发生误覆盖，先恢复文件，再同步修正提交（必要时 `git commit --amend` 仅改提交信息，或新增修复提交）。
