# AGENTS 协作说明

## 目的
本文档用于定义本仓库中人类开发者与编码代理的协作方式。

## 范围
这些说明适用于整个项目目录。

## 必读引用
- [developer_guide.md](./developer_guide.md)作为开发者文档，记载了项目定位和一些项目内概念。
- 每次有实质性变更后，在[develop_log.md](./develop_log.md)追加事实记录。
- 用户提出需要开发一个功能（设计开发方案）时，将分析结果/方案追加记入[develop_log.md](./develop_log.md)。

## 协作约定
- 当用户提供解释、澄清、偏好、约束或更正时，提取持久信息，并将其作为注释或规则记录在这里。
- 只记录用户实际陈述或清楚说明的信息。
- 不要发明用户没有提供的偏好、假设或要求。
- 更喜欢简洁，可操作的笔记，可以指导这个项目的后期工作。
- 如果一个笔记过时了，因为用户后来修改了它，更新现有的笔记，而不是保留冲突的规则。
- 变更应优雅，合理，可回溯、易审查、影响面可控。
- 仓库中文本类文件（含代码、配置、文档等）统一使用 Windows 换行符（CRLF）；处理此类文件时默认换行符可能不符合要求，不再先检查确认，直接统一修正为 CRLF。
- 版本号采用三位表示：
  - 高位每次进行复杂重构/架构优化/破坏性不兼容旧版本更新时增加
  - 中位每次新增功能，保持兼容性时增加
  - 低位每次修复漏洞时增加
  - 较高一位增加时低于其的版本号归零。

## 日志记录规则
- 每次实质性变更后，在 `develop_log.md` 追加事实记录。
- 开发日志默认只能追加记录，不能删除/改写旧内容；除非得到用户明确指示。
- 开发日志每天只记录一条时间戳，时间戳使用二级标题 `##`。
- 每条时间戳下如需新开一段，必须使用三级标题 `###`，并与上一行空行分隔；标题下内容使用 `-` 分点。
- 禁止使用“延续上个话题”作为标题；每次追加都必须创建新的 `###` 标题，标题需准确描述本次事件。
- 开发日志按“事件”拆分记录：同一时间戳下若发生两类不同性质事项（如：分析决策 vs 代码实现、功能改造 vs 环境排障），必须拆成两个二级标题（`###`）分别记录，不得混写在同一标题下。
- 开发日志每条记录只描述一个已发生事实；若某条同时包含“将要做/建议做/后续应做”与“已完成”，必须拆分或改写为纯事实表述。
- 追加开发日志前必须自检：标题是否单一主题、是否可独立回溯到对应文件改动；不满足则先拆分再写入。
- 追加开发日志时，必须通过脚本 `scripts/append-develop-log.ps1` 执行；禁止手工直接编辑 `develop_log.md` 追加内容。
- 若脚本缺失或执行失败，需先修复脚本或向用户说明阻塞原因；不得回退为手工混写。
- 假设、风险以及会影响决策或回溯的验证结果统一记录到 `develop_log.md`；禁止记录“验证通过”“无空白错误”“无格式错误”等无实际意义的通过类检查条目。
- 开发日志仅记录已发生事实，不写计划或下一步目标。
- 开发日志和开发文档不得记录本机绝对路径或本机临时环境测量值；如需描述环境、路径或体积，使用“本机 Android SDK 路径”“项目 android 原始长路径”“临时映射路径”“主要占用来自可再生成构建产物”等泛化表述。
- 签名隐私信息不得写入开发日志、开发文档或提交信息；包括 keystore 密码、密钥别名密码、证书 SHA 指纹、私有签名文件内容或其他可用于识别/复用签名材料的细节。
- 版本更新单独拥有一条时间戳，单独记录一条开发日志。

## 提交规则
- 提交时遵循提交规范：
- summary里填写[性质] 内容概括。description里填写逐文件变化说明。
  - [ver]：版本更新（版本号提升单独提交，使用该前缀）
  - [doc]：文档更新
  - [fix]：漏洞修复
  - [feat]：应用内用户可见的新功能实现，或应用底层用户不可见的新功能实现
  - [dev]：开发流程、构建/发布链路、工程脚本与环境能力改造（非应用内用户可见功能）
- 当有[fix]/[feat]/[dev]改动时，对应的[doc]改动应一并提交。
- [fix]/[feat]/[dev]改动应根据实际改动内容判断分几次，而不是所有[fix]/[feat]/[dev]一次性提交。
- 提交时采用英文
- git工作区存在多项改动时，根据改动内容分成合适的次数提交，而不是一次性提交。
- 当用户要求“仅提交部分文件/部分改动”时，必须执行以下防错步骤：
  - 提交前先执行 `git status --short` 与 `git diff -- <目标文件>`，明确提交边界。
  - 仅使用显式 `git add <文件列表>` 暂存目标文件，不使用全量暂存。
  - 如果目标文件中混有无关改动，优先使用 `git add -p <目标文件>` 仅暂存目标 hunk；不得通过修改/删除工作区内容来“排除提交”。
  - 仅当 `git add -p` 无法安全表达边界时，才允许“备份-最小化提交-恢复”流程；恢复后必须再次执行 `git diff -- <目标文件>` 确认无关改动已回到未提交状态。
  - 提交后执行 `git show --name-only --stat -1` 与 `git status --short` 复核；若发现误覆盖/误提交，立即修复并向用户说明。
  - 若发生“为排除提交而误改工作区内容”的事故，必须在同一会话立即恢复该内容，并在 `develop_log.md` 追加事故与修复事实记录。

## 用户已明确偏好（持久）
- 用户主要通过应用页面实际效果验证开发结果；涉及重构方案、抽组件或内部结构调整时，应优先按用户可见页面、入口、弹窗、列表、按钮、图表和验证路径说明，不用代码术语作为主要表达。
- 当前项目基于 Expo 开发，用户希望减少“仅依赖 Expo Go 扫码”带来的设备尺寸测试限制。
- 用户希望构建流程尽量在项目仓库内产出安装包/发布物，以便后续 release 管理与分发。
- 当前阶段开发与测试仅考虑 Android，不纳入 iOS。
- 用户希望在 bare workflow 下保持 `expo.version` 与 `ios/android runtimeVersion` 自动统一，不采用手工分别维护。
- 在 Windows 本机进行 Android 构建时，采用“构建前临时 `subst` 映射短路径、构建后无论成功失败都取消本次创建映射”的流程；构建前若目标盘符已存在，无论是否映射到当前项目，都直接失败，不自动删除，避免误删用户映射或干扰并发构建。
- Android 安装包只有准备发布或分发时才需要复制归档并重命名；普通构建不要求每次复制重命名。
- Android 安装包发布/分发归档命名格式为 `MinimalistWeaponEnhancementCalendar-v<语义版本>-android-<yyyyMMdd>.apk`，归档位置为 `dist/`，归档来源必须是 Release APK。
- Android Release APK 使用本机私有 release 签名；`android/app/debug.keystore` 仅用于 Debug，Release 构建不得使用 `signingConfigs.debug`。
- Android Release 签名配置读取 `android/keystore.properties`，私有签名文件为 `android/app/release.keystore`；两者必须保持 Git 忽略，禁止提交到开源仓库。
- 正式发布版本之后开发的未发布功能，必须先记录到版本记录页的 `Unreleased` 节点；正式发布时再把该节点版本号改成对应的正式版本号。
- 版本记录页里 `Unreleased` 前面不加 `v`。`Unreleased` 本身不天然作为特殊版本；只有当前正在开发的版本有明确主功能时，才可先用与“新增、修复、优化”同级的标题单独介绍该功能，再显示“新增、修复、优化”分组。
- 版本记录页的专题功能介绍应围绕该功能本身描述能力和使用结果，不在每条说明中重复使用“新增……”句式。
- 版本记录页面只记录用户能感受到的变化，不记录内部实现细节。
- 版本记录页面记录的是相对上一版本的最终变化；同一版本开发过程中发生的中间调整、反复修正或临时状态不写入版本记录，只保留最终态说明。
- `Unreleased` 与正式版本节点遵循同一规则：只记录相对上一正式版本的最终用户可感知变化，不记录本轮开发过程中的开发记录、调试记录、临时方案、回滚、反复修正或内部实现调整。
- 同一版本内同一功能若属于“新增”，不得再把该功能开发过程中的完善、优化、修复拆成同版本的“优化”或“修复”条目；应合并为新增功能最终态描述。只有相对上一版本已经存在的功能发生变化时，才写入“优化”或“修复”。
- 版本记录不是开发日志；任何仅用于回溯开发过程、验证过程、事故处理或实现细节的内容必须记录到 `develop_log.md`，不得写入版本记录页。
- 每个版本节点内按“新增、修复、优化”分组展示；某类无内容时不显示该分类；初始版本不显示“新增”分类标题。
- 软件介绍页面只介绍应用定位、记录类型和用户可见功能，不暴露权限、刷新、数据结构或实现规则等不必要细节。
- 每次功能改动，新功能开发后，在提交前需要检查软件介绍、使用帮助、隐私政策、版本记录，以及本地开发文档AGENTS、developer_guide、develop_log，七个地方是否都统一且符合软件实际。
- 开发黑名单功能时，只有改动无法直接同步到在线模拟器时，才需要重新安装到模拟器验证。
- 关于入口位于主界面右下角“更多”菜单中，并始终作为该菜单最后一项。
- 黑名单完整应用列表只在“设置黑名单应用”页面读取；应用启动后首次进入该页面正常全量读取并缓存，本次应用运行期间后续读取优先使用缓存，点击该页面右下角刷新按钮时重新全量读取。
- “设置黑名单应用”页面顶部“已选择 N 个应用”状态框固定在顶部并保持原有框样式，状态框下方提供搜索栏，搜索栏下方左侧用小字提供“全部 / 已选”列表切换；只滚动下方应用列表；应用列表中英文按拼音首字母混排，无法归入 A-Z 的应用归入末尾 #；应用列表框与上方状态框、搜索框同宽；首字母快速滑动条位于应用列表框外并贴近屏幕右侧边缘，触碰时高亮当前命中字母；刷新按钮位于右下角，位置语义接近日视图/月视图的回到今日/回到本月按钮；刷新按钮自身不显示旋转动画，加载和刷新统一使用列表区域加载动画。
- 黑名单应用选择变化在“设置黑名单应用”页面内通过 Android 原生黑名单数据事务实时保存，避免未正常退出导致设置丢失；黑名单应用主色随该页面的黑名单应用更新流程保存；黑名单主页只读取使用情况和已保存主色，不自行读取完整应用列表或提取主色；查看使用时间段页面只读取已保存使用时间段，不读取完整应用列表或提取主色。
- 黑名单应用选择以生效时间段保存；新增黑名单应用不会回填加入前的历史记录，取消黑名单或应用卸载只结束当前生效时间段，不删除应用档案或历史使用记录。
- 设置黑名单应用页面刷新当前可启动应用列表时，当前生效黑名单中已卸载或不可启动的应用自动取消选择并标记为已卸载；历史使用记录页仍可显示和筛选这些应用的历史记录。
- “查看使用记录”的全部记录详情中，每条使用时间段记录前方显示对应应用图标。
- “查看使用记录”的全部记录详情和单个应用记录详情中，顶部汇总统计固定在顶部，只滚动下方记录；汇总框下方、记录列表上方提供“今日 / 7天 / 30天”三个范围按钮，默认显示 7天，点击对应按钮切换记录和汇总统计；右侧筛选按钮打开筛选弹窗，可设置日期范围。自定义日期范围确认后，原范围按钮位置显示当前日期范围；再次打开筛选弹窗时显示当前筛选态，存在筛选时左侧操作为清除筛选。全部记录详情的筛选弹窗顶部按屏幕宽度显示当前选中应用图标预览，最右侧编辑按钮可打开应用显示筛选弹窗，在已显示和未显示应用图标之间点击移动；单个应用详情的筛选弹窗顶部居中显示当前应用图标。
- 黑名单主页每次进入时在后台静默读取最近三天使用记录，不显示动画或弹窗；下拉刷新读取同样的最近三天使用记录；“按日期读取记录”按钮打开起止日期弹窗，点击起止日期框后复用自定义统计页的 DatePicker 页面选择逻辑和顺序校验，返回后继续显示读取弹窗，确认后读取并保存所选日期范围内的使用记录，完成提示同时显示请求读取范围和实际读取到的记录覆盖范围。
- 使用记录辅助开启后的晚间定时刷新与黑名单主页下拉刷新使用同样的最近三天读取范围，并必须将读取到的黑名单应用使用时间段实质保存到本地，而不是只写刷新状态。
- 黑名单同一应用碎片记录合并不得跨越其他黑名单应用使用时间段；只有间隔不超过 2 分钟且中间没有其他黑名单应用使用时才合并。
- 黑名单主页在“查看使用时间段”入口下方提供“今日 / 7天 / 30天 / 热图”按钮组，按按钮切换显示当日、最近 7 天、最近 30 天统计图和自适应热力图；不使用自然周或自然月作为今日、7 天、30 天主页统计图范围。
- 黑名单主页热力图按 GitHub 贡献图形式展示使用时长，一列 7 个格子表示周日到周六，横轴只显示月份；图表按屏幕决定外框和绘图区宽高，再由一列 7 格约束计算格子大小，并由格子大小和绘图区宽度决定可显示列数。
- 黑名单主页热力图无使用格子使用白色或中性底色与中性边框，避免和黄橙色使用强度混淆。
- 黑名单主页热力图按每日使用分钟数连续渐变映射颜色，映射范围为 `0` 到当前可见范围单日最大分钟数向上取整到 10 的值；底部偏右显示对应的连续渐变图例，图例需与右侧边缘保持内缩，避免最大值位数较多时超出绘图区。
- 黑名单最近 7 天柱状图触摸时按坐标命中日期，框选对应柱子并在旁边显示时长，不使用 30 天趋势图那种垂直虚线。
- 黑名单最近 30 天趋势图中，连续多天无使用记录时只显示这段连续空心点的首尾点，中间仅保留趋势线；触摸图表时显示对应日期的竖向参考线、点位和分钟数。
- 黑名单最近 7 天柱状图和最近 30 天趋势图的纵坐标刻度标签显示在图表左侧。
- 黑名单最近 7 天柱状图和最近 30 天趋势图通过横轴数据范围左右内缩避免纵坐标刻度贴近绘图，不通过扩大绘图区左侧预留宽度实现。
- 黑名单功能记录的实际含义归入“观看教程”，因此黑名单主页、黑名单应用页、使用记录页及图表中的按钮、筛选、选中态、刷新入口、弹窗内容和非白色视觉强调统一使用观看教程对应的黄橙色系；弹窗外界遮罩属于中性行为，不使用黄橙色系。
- 黑名单使用记录动态计算观看教程自动记录次数，并作为日历展示和统计中观看教程次数的下限；自动记录次数不单独持久化，从已保存的黑名单使用时间段读取时计算；黑名单主页顶部提示框说明该联动，提示框内信息图标弹窗显示完整计算规则。
- 观看教程自动记录次数基于既有 2 分钟黑名单碎片合并后的使用时间段继续聚合；所有黑名单应用使用时间段按时间排序，前一段结束到后一段开始间隔不超过 1 小时视为同一次，允许跨日聚合，跨日聚合段按开始时间所在日期计入；同一次自动记录的已保存使用时长合计少于 5 分钟时不计入自动记录次数。
- 用户通过 UI 修改的是观看教程总次数；次数不能低于自动记录下限。长按递减只能减到自动记录下限，到下限后继续取消时振动并提示该记录由黑名单应用使用记录自动生成，当前次数不能低于自动记录次数；提示弹窗提供查看记录入口，可跳转到使用时间段的全部记录详情并筛选到触发提示的日期。
- 日历视图内触发的自动记录不可取消提示使用普通应用弹窗样式，不使用黑名单功能的黄橙色系。
- 用户希望新的数据结构从整个软件数据模型层面统一管理，第一级区分打卡数据、黑名单数据和用户配置数据；影响记录语义和读取范围的黑名单应用状态归入黑名单数据，纯界面偏好归入用户配置数据。
- 导入、导出和分享对外使用完整应用数据 JSON，覆盖打卡数据、黑名单数据和用户配置数据；旧版仅含打卡记录的 JSON 导入时只更新打卡数据，不清空现有黑名单数据或用户配置。
- 日视图、月视图、年视图和日期选择页支持点击顶部日期文字弹出滚轮选择框快速切换日期、月份或年份；日期选择页进入某一月后点击顶部年月只能切换年月，不能直接选定具体日期。
- 年度统计顶部年份文字支持点击弹出滚轮选择框快速切换年份。
- 日期选择页从年份选择进入月份日期选择、从月份日期选择返回年份选择时，使用与外层年份选择页进出一致的默认进入/退出动画，不使用横向平移或纯透明淡入淡出。
- 底部栏左侧日历按钮按日视图、月视图、年视图轮回切换，并以“查看日视图 / 查看月视图 / 查看年视图”显示下一步目标；月视图和年视图中间主按钮不提供返回日视图功能。
- Android 原生构建中，问题日志仅在 JS 未捕获异常、未处理 Promise、Android 未捕获异常或主线程长时间无响应时写入应用专属外部文件目录的 `diagnostic-logs` 文件夹；正常运行不持续记录日志，设置页仅在检测到日志文件时显示打开问题日志文件夹入口。
- 极简备忘录里的内容统一称为“笔记”；“极简备忘录”仅作为外层入口、模式或应用名称使用。
- 安全锁入口位于设置页数据管理分组下方、黑名单权限管理分组上方；点击入口进入安全锁功能页面，不直接开启安全锁。
- 设置页三个权限类开关和安全锁页面开关使用同样的蓝色视觉样式；这些开关只作为当前实际状态展示，不在触摸瞬间改变显示，状态只在系统设置返回、弹窗关闭或实际启停完成后刷新。
- 设置页精确定时权限开关显示系统精确定时权限本身是否授权，不用电池优化例外等其他能力替代。
- 安全锁默认关闭，开启前必须显示强告知弹窗并要求输入指定确认内容；开启后启动应用先进入本地极简备忘录，用户在新建笔记正文中输入与密码完全一致的内容并保存后进入真实记录应用，匹配成功的内容不保存为笔记。
- 安全锁密码不保存明文，仅保存带随机盐的哈希校验信息；只有新建笔记保存时进行密码校验，编辑已有笔记不触发进入内部应用。
- 极简备忘录是真实本地笔记功能，支持笔记标题、正文、分类、搜索、手动保存、导入和导出；笔记标题未填写时使用正文第一句话前若干字生成默认标题。
- 极简备忘录分类通过较大的可滚动弹窗查看和选择；新建分类需输入分类名称并设置分类颜色，分类颜色采用少量预设色加 RGB 自定义并显示 HEX 值。
- 极简备忘录外层设置页的数据管理只处理笔记数据；完整应用数据导入和导出仍保留在进入内部应用后的设置页。
- 笔记导入采用合并导入；同一笔记仅由唯一 ID 判断，同 ID 冲突时提供替换、跳过、存为副本，不按标题或正文去重；同名分类导入时可选择导入当前同名分类或新建副本分类，副本名按 `(1)`、`(2)` 递增。
- 备忘录模式桌面名称显示为“极简备忘录”，图标和启动页使用内置备忘录视觉；关闭安全锁或重置应用后恢复原桌面名称、图标和启动页。
- 进入内部应用后关闭安全锁不需要再次验证密码，只需弹窗二次确认；安全锁开启强告知强校验，关闭保持易关闭。
- 安全锁不设计免密恢复真实应用入口；极简备忘录设置页提供重置应用按钮，忘记密码时可重置应用。重置会清空打卡数据、黑名单数据、用户配置、笔记内容、问题日志和应用可控缓存，关闭安全锁并恢复崭新安装状态。
- 重置应用前仅引导关闭当前已开启的黑名单使用记录权限和电池优化权限；当前未开启的权限不引导关闭；不专门引导关闭精确定时权限，重置后应用内相关功能和本地状态恢复为未启用。

## Windows Android 构建执行流程（强约束）
- 适用范围：所有本机 Android Gradle 构建与安装流程。
- 禁止事项：禁止直接在项目 android 原始长路径执行 `gradlew` 系列命令。
- 标准命令（仓库根目录）：
  - 构建 Debug APK：`npm run android:build:debug:tempmap`
  - 构建 Release APK：`npm run android:build:release:tempmap`
  - 构建并安装到模拟器/设备：`npm run android:install:debug:tempmap`
- 执行语义（必须满足）：
  - 步骤 1：每次构建前先检查目标临时盘符（默认 `M:`）是否已存在；若已存在，无论是否映射到当前项目，都直接失败，不自动删除。
  - 步骤 2：仅当目标盘符不存在时，创建本次构建专用临时映射。
  - 步骤 3：仅在临时映射路径下的 android 目录内执行 Gradle 任务。
  - 步骤 4：构建结束后无论成功或失败都执行取消本次创建的映射（`subst M: /D`）。
  - 步骤 5：构建结束后校验 `subst` 输出不包含 `M:`。
  - 步骤 6：下一次构建必须重新创建映射，不复用旧映射；若发现旧映射残留，先人工确认没有并发构建占用后再清理。
- 实现载体：`scripts/android-build-tempmap.ps1`（`try/finally` 保证清理）。
- 发布/分发归档：仅在准备发布或分发安装包时，将 `android/app/build/outputs/apk/release/app-release.apk` 复制到 `dist/` 并按 `MinimalistWeaponEnhancementCalendar-v<语义版本>-android-<yyyyMMdd>.apk` 命名；普通构建不执行该归档步骤，禁止将 `debug/app-debug.apk` 作为发布或分发归档来源。
- Release 签名：Release APK 必须使用 `android/keystore.properties` 指向的本机私有签名；缺少私有签名配置时不得回退为 debug 签名。

- 当前 AVD 尺寸语义约定：`AVD_2640x1200` 为手机尺寸默认机型，`AVD_2560x1600` 为平板尺寸机型。
- 视觉上占满整页、用户会理解为独立页面的界面，默认应实现为 Screen 文件并注册到导航；仅顶部栏、弹窗、滑条、列表项等可复用局部界面放在 components。
